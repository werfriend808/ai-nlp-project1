# A+B / C+D 연결 테스트 기록 (Day2 오후, 1→6단계 전체 자동 연결)

- 대상: `agent/pipeline/batch_runner.py` (A의 1·2단계 + B의 3단계 + D의 4단계 + C의 5·6단계 전부 실제 함수로 연결)
- 결과: **1→2→3→4→5→6단계 전체 자동 연결 성공** (실제 HCX API + 실제 KOSIS API 호출 포함, 2개 시나리오 모두 통과).
- 처음 실행했을 때는 B의 `table_catalog.json`과 C의 `table_params.json`이 겹치는 표가 하나도 없어서 5단계에서 전부 막혔음 (아래 이슈 1). `table_params.json`에 B의 카탈로그와 실제로 겹치는 표 2개(`DT_1J22003` 소비자물가지수, `DT_1DA7001S` 성별 경제활동인구총괄)를 KOSIS API로 직접 조사해서 추가한 뒤, 두 시나리오 모두 6단계까지 자동으로 도달함.

## 실행 방법

프로젝트 루트에서 실행 (`.env`에 `HCX_API_KEY`, `KOSIS_API_KEY` 둘 다 필요 — 실제 API 호출):

```
python -m agent.pipeline.batch_runner
```

Windows 콘솔에서 한글이 깨지면 `PYTHONUTF8=1`을 앞에 붙여서 실행.

### 시나리오 2 — 소비자물가 기사
```
[1단계] label=True, score=0.95

[3단계] "소비자물가지수(2020=100)"(DT_1J22003) 자동 매칭, score=1.000
```

**주의(계산 기준 관련 캐비앗)**: 기사 원문은 "전년 **동월** 대비 2.2%"(월 단위 비교)라고 주장하는데, 지금 파이프라인은 D의 `period` 슬롯이 연도 단위("2024")로만 채워지기 때문에 `prdSe=Y`(연간 평균) 기준으로 2023→2024를 비교해서 2.3%가 나왔습니다. 기사 주장(월 단위 2.2%)과 검증값(연간 단위 2.3%)은 **비교 기준 자체가 달라서** 이 둘을 그대로 "불일치"로 판정하면 안 됩니다 — 7단계(판정) 설계 시 월/연 단위를 맞추거나, 최소한 "비교 기준이 다르다"는 걸 판정 근거에 명시해야 합니다.

## 발견된 이슈

### 1. (부분 해결) B의 `table_catalog.json`과 C의 `table_params.json`이 원래 겹치는 표가 하나도 없었음
- B의 카탈로그: `DT_1J22003`, `DT_1DA7001S`, `DT_1B04005N`, `DT_200Y102`, `DT_1R11006_FRM101`, `DT_30404_B012`, `DT_1B8000G` (7개)
- C의 파라미터(원래): `DT_1EA1019`, `DT_1DA7102S` (2개) — **교집합 0개**였음.
- 이번에 `DT_1J22003`(소비자물가지수), `DT_1DA7001S`(성별 경제활동인구총괄)를 KOSIS API로 직접 조사해서 `table_params.json`에 추가함 → 이제 교집합 2개, 위 두 시나리오는 6단계까지 자동 도달.
- **아직 남은 것**: B 카탈로그의 나머지 5개(`DT_1B04005N` 인구, `DT_200Y102` 경제성장, `DT_1R11006_FRM101` 무역, `DT_30404_B012` 부동산, `DT_1B8000G` 출생/사망/혼인)는 여전히 `table_params.json`에 없음. C 담당자가 이어서 조사해야 함.

### 1-1. (신규 발견) `DT_1DA7001S`는 itmId가 여러 개라 "어떤 지표인지" 자동으로 못 고름
`DT_1DA7001S`는 실업률(T80) 외에도 취업자(T30)/고용률(T90)/경제활동참가율(T60) 등 8개 지표를 한 표 안에 담고 있음. 지금은 B의 매칭 키워드 중 가장 흔한 "실업률"(T80)로 고정해뒀는데, 그래서 시나리오 1의 두 번째 주장("취업자 수는 46개월 만에 감소로 전환했다")도 실제로는 T30(취업자)을 조회해야 맞는데 T80(실업률)으로 잘못 조회되고 있음 — 계산 자체는 성공하지만 **엉뚱한 지표를 검증하고 있는 상태**. claim.sentence에서 어떤 itmId를 골라야 하는지 판단하는 로직이 3~5단계 사이에 필요함 (미해결).

### 1-2. (신규 발견) period 단위(연/월) 불일치로 기사 주장과 다른 기준으로 검증됨
소비자물가 시나리오에서 기사는 "전년 **동월** 대비 2.2%"(월 단위)라고 주장하는데, D의 `period` 슬롯이 연도만 채우기 때문에 실제로는 연간 평균 기준 2023→2024 비교(+2.3%)로 검증됐음. 값이 비슷해서 우연히 그럴듯해 보이지만, **비교 기준 자체가 다름**. 7단계(판정) 설계 시 반드시 감안해야 함.

### 2. (확인됨, 설계상 알려진 갭) D의 `REQUIRED_SLOTS`가 표와 무관하게 고정되어 있어, region이 없는 표에도 항상 지역을 되묻는다
- `clarify_rules.REQUIRED_SLOTS = ["period", "region", "calc_type"]`는 모든 claim에 동일하게 적용됨.
- 그런데 `DT_1DA7102S`(실업률표)는 `table_params.json`상 `region` 차원 자체가 없음(gender/age만 있음). 그래도 clarify는 무조건 "어느 지역 기준인가요?"를 물어봄.
- `interfaces.py` 설계상 `required_slots`는 `TableCandidate`(3단계 결과, 표마다 다름)에서 나와야 하는데, 지금 D의 로직은 3단계 결과와 무관하게 고정 리스트를 씀. → 이번 연결 테스트로 실제 재현됨(시나리오 1, 2 모두 region 되묻기 발생).
- 근본적으로 고치려면 `get_next_clarify_step()`이 `TableCandidate.required_slots`를 인자로 받도록 D+B가 함께 인터페이스를 조정해야 함. 지금은 손 안 댐(팀 합의 필요 사항).

### 3. (참고) `calc_type`(D, 6단계 계산 종류)과 `claim_type`(A, 2단계 주장 유형)은 서로 다른 값 체계
- A가 뽑는 `claim_type`: `["규모", "증감률", "비교", "전망"]` (+가이드 PDF엔 "역대기록"도 언급, `interfaces.py`엔 없음 — B 로그에서도 지적한 사항).
- D의 `fill_slots`가 발화에서 추출하는 `calc_type`은 자유 텍스트에 가까움 (실행 결과에서 "실업률", "전년 동월 대비" 같은 값도 나옴 — `interfaces.py`의 `CalcType = Literal["합계","비율","증감","증감률"]`에 없는 값들). 지금 `run_stage_5_6`은 `calc_type in ("증감","증감률")`만 특별 처리하고 나머지는 전부 "단순 조회"로 취급하므로 크래시는 안 나지만, 사용자가 "합계"/"비율"을 요청하는 케이스는 이번 파이프라인에서 실질적으로 처리 못 함.

## 커버 안 된 것 (다음에 필요하면 추가)
- B 카탈로그 7개 중 5개는 여전히 `table_params.json`에 없음 (인구/경제성장/무역/부동산/출생 계열 — 이슈 1 참고).
- `DT_1DA7001S`의 itmId(어떤 지표를 조회할지) 자동 선택 로직 없음 (이슈 1-1).
- period 연/월 단위를 기사 주장과 맞추는 로직 없음 (이슈 1-2).
- `합계`/`비율` calc_type에 대한 자동 처리는 미구현 (여러 카테고리 값을 모아 `compute_sum`/`compute_ratio`에 넘기는 로직 필요, 지금은 `증감`/`증감률`만 자동 처리).
- 8단계(설명 생성)는 이번 범위 밖.
