## Day3 — 7·8단계(judge/explainer) 통합 결과

D가 `agent/verdict/judge.py`(7단계), `agent/explain/explainer.py`(8단계)를 완성하고 `batch_runner.py`에 `run_stage_7_8`로 연결함 (`run_stage_5_6`이 `ComputedResult`를 반환하도록 바뀌고, judge/explain 각각 try/except로 감싸서 실패해도 배치가 안 죽게 처리됨). 10개 시나리오로 재실행한 결과, **1~8단계 전체가 예외 없이 끝까지 도는 것은 확인됨.**

다만 10개 중 실제로 7·8단계까지 도달한 건 4건(시나리오 1의 주장 2개, 시나리오 2, 6, 8)뿐이고 — 나머지는 4단계 되묻기 미해결(이슈 1-5) 또는 5단계 실패(이슈 1-3)로 그 전에 멈춤 — 그 4건을 보니 **judge.py의 판정 로직 자체에 심각한 문제**가 있음 (아래 이슈 4).

### judge.py 실행 예시 (문제 재현)
```
주장: "청년층 취업자 수는 46개월 만에 감소로 전환했다."
[6단계] ComputedResult(calc_type='증감률', raw_value=3.7, unit='%', period='2023~2024')
[7단계] Verdict(verdict='불일치', gap_type='수치',
        reason='기사 수치(-46.0)와 통계 계산값(3.7%) 차이가 49.70%p로 허용 오차를 크게 초과함')
```
"46개월"의 "46"을 마치 증감률(-46%)인 것처럼 뽑아서 비교함 — "46개월 만에"는 기간 표현이지 통계 수치가 아님.

```
주장: "지난달 소비자물가가 전년 동월 대비 2.2% 오른 것으로 나타났다."
[6단계] 단순 조회 → raw_value=116.61, unit='2020＝100'
[7단계] reason='기사 수치(2.2)와 통계 계산값(116.612020＝100) 차이가 114.41%p로 ...'
```
기사의 "2.2%"(변화율)와 통계의 "116.61"(원본 지수 레벨)을 비교 — 애초에 종류가 다른 값이라 비교 자체가 무의미함. `reason` 문자열도 `raw_value`와 `unit` 사이에 구분자가 없어 `"116.612020＝100"`처럼 붙어버리는 포맷 버그도 있음.

반면 시나리오 8("출생아 수는 23만8천명")은 기사가 규모(절대값)를 주장했고 계산값도 규모(238,317명)라서 종류가 맞아떨어져 "일치"로 정상 판정됨 — 즉 judge.py의 산술 비교 로직 자체는 문제없고, **비교하려는 두 숫자가 같은 종류(수준 vs 변화율, 절대값 vs 기간 등)인지 확인하지 않는 것**이 근본 원인.

## 발견된 이슈

### 1. (해결) B의 `table_catalog.json`과 C의 `table_params.json`이 원래 겹치는 표가 하나도 없었음
- B의 카탈로그: `DT_1J22003`, `DT_1DA7001S`, `DT_1B04005N`, `DT_200Y102`, `DT_1R11006_FRM101`, `DT_30404_B012`, `DT_1B8000G` (7개)
- C의 파라미터(원래): `DT_1EA1019`, `DT_1DA7102S` (2개) — **교집합 0개**였음.
- 7개 전부 KOSIS API로 직접 조사해서 `table_params.json`에 추가함 → **이제 교집합 7/7 (전부 커버)**.
- `build_kosis_slots` + 실제 `KosisApiClient`로 5개(`DT_1J22003`, `DT_1DA7001S`, `DT_1B04005N`, `DT_1R11006_FRM101`, `DT_1B8000G`)는 `{"period":"2024","region":"전국"}` 수준의 generic slots만으로 정상 조회 확인함 (인구 51,217,221명 / 수출액 683,609,488천달러 / 출생아수 238,317명 등).
- 나머지 2개는 아래 이슈 1-3 참고 (표 자체의 시간 단위 문제로 여전히 자동 연결 안 됨).

### 1-1. (신규 발견) `DT_1DA7001S`, `DT_1R11006_FRM101`, `DT_1B8000G`는 한 표 안에 지표가 여러 개라 "어떤 지표인지" 자동으로 못 고름
- `DT_1DA7001S`: 실업률(T80)/취업자(T30)/고용률(T90)/경제활동참가율(T60) 등 8개. 지금은 "실업률"로 고정 → 시나리오 1의 "취업자 수" 주장도 실제론 T30을 조회해야 하는데 T80으로 잘못 조회됨.
- `DT_1R11006_FRM101`: 수출액/수입액 2개. 지금은 "수출액"으로 고정 → "수입" 관련 주장은 자동으로 못 맞음.
- `DT_1B8000G`: 출생아수/합계출산율/혼인건수/이혼건수 등 15개 항목이 objL2 축에 있음. 지금은 "출생아수"로 고정 → "혼인 건수" 같은 주장은 자동으로 못 맞음.
- 공통 원인: B의 3단계 매핑이 table_id까지만 골라주고, "그 표 안에서 어떤 지표(itmId/objL2)를 볼지"는 안 정해줌. claim.sentence를 보고 어떤 지표인지 좁혀주는 로직이 3~5단계 사이에 필요함 (미해결, 3개 표 공통 이슈).

### 1-2. (신규 발견) period 단위(연/월) 불일치로 기사 주장과 다른 기준으로 검증됨
소비자물가 시나리오에서 기사는 "전년 **동월** 대비 2.2%"(월 단위)라고 주장하는데, D의 `period` 슬롯이 연도만 채우기 때문에 실제로는 연간 평균 기준 2023→2024 비교(+2.3%)로 검증됐음. 값이 비슷해서 우연히 그럴듯해 보이지만, **비교 기준 자체가 다름**. 7단계(판정) 설계 시 반드시 감안해야 함.

### 1-3. (신규 발견, 미해결) `DT_200Y102`(경제성장), `DT_30404_B012`(부동산)는 연간 데이터 자체가 없음
- `DT_200Y102`(GDP 성장률): **분기(Q) 단위 전용**. `prdSe=Y`로 조회하면 "데이터가 존재하지 않습니다" 에러. period 형식도 `YYYYQQ`(예: `202404`)라서 D가 채우는 `"2024"`를 그대로 넘기면 무조건 실패함.
- `DT_30404_B012`(주택매매가격지수): **월(M) 단위 전용**. 마찬가지로 `prdSe=Y`는 데이터 없음, period 형식은 `YYYYMM` 필요.
- 실제로 `build_kosis_slots("DT_200Y102", {"period":"2024",...}, ...)` → `KosisApiError: [30] 데이터가 존재하지 않습니다` 로 재현됨 (부동산표도 동일 에러).
- 위 이슈 1-2(CPI)는 그나마 연간 폴백이 있어서 "다른 기준으로라도" 값이 나왔는데, 이 둘은 **연간 폴백 자체가 없어서 100% 실패**함. D의 `slot_filler`가 분기/월까지 뽑아내도록 프롬프트를 바꾸거나, 파이프라인에서 연도→최근 분기/월로 변환하는 로직이 있어야 풀림. 지금은 두 표 다 `table_params.json`에 `_important_note`로 명시만 해두고 미해결로 남김.

### 1-4. (해결) 3단계(B)가 "출생아 수" 주장을 무역표로 잘못 매칭함
10개 시나리오 배치에서 발견. 시나리오 8 "작년 출생아 수는 23만8천명으로 전년보다 늘어난 것으로 나타났다" → `search_and_rerank()`의 최상위 후보가 `DT_1B8000G`(출생/사망/혼인, 정답)가 아니라 **`DT_1R11006_FRM101`(국가별 수출액, 수입액)**로 나왔었음. 이 잘못된 표로 그대로 진행돼서 5·6단계까지 "정상적으로" 값(+8.1%)이 계산됐는데, **이건 출생아 수와 전혀 무관한 수출액 증감률**이었음 — 표 매칭이 틀렸는데 뒷단계는 아무 이상 없이 성공한 것처럼 보이는 게 더 위험함(조용히 틀린 결과를 냄).

**원인**: `keyword_search`의 `SYNONYMS` 사전에 "출생아"라는 키 자체가 없었음 (기존엔 "출산"/"저출생"만 있었는데 "출생아 수"라는 실제 기사 표현은 이 두 단어를 부분 문자열로도 포함하지 않음 — "출생아"엔 "산"이 없고 "저출생"도 부분 매칭 안 됨). 그래서 keyword_search가 매칭에 실패했고, `embedding_search`의 더미(해시 기반) 벡터 노이즈가 순위를 결정하면서 엉뚱한 표가 1위로 올라온 것 — keyword가 못 잡으면 embedding 노이즈가 순위를 정하게 되는 구조적 약점이 실제로 재현된 사례.

**수정**: `keyword_search.py`의 `SYNONYMS`에 `"출생아": ["출생아수", "합계출산율", "출생률"]` 추가. 재검증 결과 `keyword_search` 단독으로 `DT_1B8000G` score=1.0으로 정확히 매칭되고, 배치 재실행 시 시나리오 8이 6단계까지 정상 도달 (+3.6%, 실제 KOSIS 값 230,028명→238,317명). `test_mapping.py` 재실행으로 회귀 없음도 확인 (keyword 11/12, reranker 12/12 유지).

### 1-5. (신규 발견) 여러 시나리오가 "지난해" 표현을 인식 못 해 4단계에서 막힘
시나리오 3·5·9가 전부 "지난해"라는 표현이 포함된 문장인데 `fill_slots` 1차 시도에서 `period`가 안 채워짐. D 모듈 로그(`d_module_test_log.md`)에 이미 "작년"만 단독으로 오면 실패하는 이슈가 기록돼 있는데, "지난해"는 그 로그에 없던 **새로운 실패 표현**임 — `RELATIVE_YEAR_OFFSET`에 정의는 돼 있는데도 LLM 추출 단계에서 놓침. 게다가 이번 배치의 `clarify_reply`("전국 기준으로 알려줘")에도 연도/계산종류 키워드를 안 넣어서 2차 시도까지도 채워지지 않아 되묻기 루프가 안 끝남 — 이건 실제 대화형 시스템이라면 사용자가 "작년이요"처럼 재답변하면서 자연히 풀릴 상황이라, 이번 배치 스크립트의 `clarify_reply` 설계 한계이기도 함. 다만 "지난해"가 "작년"과 같은 뜻인데도 LLM이 놓치는 것 자체는 D 쪽 few-shot 보강이 필요한 실제 이슈임.

### 2. (확인됨, 설계상 알려진 갭) D의 `REQUIRED_SLOTS`가 표와 무관하게 고정되어 있어, region이 없는 표에도 항상 지역을 되묻는다

- `interfaces.py` 설계상 `required_slots`는 `TableCandidate`(3단계 결과, 표마다 다름)에서 나와야 하는데, 지금 D의 로직은 3단계 결과와 무관하게 고정 리스트를 씀. → 이번 연결 테스트로 실제 재현됨(시나리오 1, 2 모두 region 되묻기 발생).
- 근본적으로 고치려면 `get_next_clarify_step()`이 `TableCandidate.required_slots`를 인자로 받도록 D+B가 함께 인터페이스를 조정해야 함. 지금은 손 안 댐(팀 합의 필요 사항).

### 3. (참고) `calc_type`(D, 6단계 계산 종류)과 `claim_type`(A, 2단계 주장 유형)은 서로 다른 값 체계
- A가 뽑는 `claim_type`: `["규모", "증감률", "비교", "전망"]` (+가이드 PDF엔 "역대기록"도 언급, `interfaces.py`엔 없음 — B 로그에서도 지적한 사항).
- D의 `fill_slots`가 발화에서 추출하는 `calc_type`은 자유 텍스트에 가까움 (실행 결과에서 "실업률", "전년 동월 대비" 같은 값도 나옴 — `interfaces.py`의 `CalcType = Literal["합계","비율","증감","증감률"]`에 없는 값들). 지금 `run_stage_5_6`은 `calc_type in ("증감","증감률")`만 특별 처리하고 나머지는 전부 "단순 조회"로 취급하므로 크래시는 안 나지만, 사용자가 "합계"/"비율"을 요청하는 케이스는 이번 파이프라인에서 실질적으로 처리 못 함.

### 4. (신규 발견, 중요, 미해결) `judge.py`가 기사 문장에서 뽑은 숫자와 계산값의 "종류"를 구분하지 않고 비교함
- claim.sentence에서 숫자를 추출할 때 그 숫자가 무엇을 의미하는지(수준값/변화율/기간/연도 등) 구분하지 않고 그대로 `computed.raw_value`와 뺄셈 비교함.
- 실제 재현 사례 (Day3 7·8단계 통합 결과 섹션 참고):
  - "46개월 만에"의 "46" → 증감률 -46%로 오인, 3.7%와 비교해 "불일치"
  - "3년 만에"의 "3" → 3.0으로 오인, 8.1%와 비교해 "불일치"
  - 기사의 변화율(%)과 계산된 원본 지수 레벨을 그대로 비교 (종류 자체가 다름)
- 부가 버그: `reason` 문자열에서 `raw_value`와 `unit`을 이어붙일 때 구분자가 없어 `"116.612020＝100"`처럼 붙어버림 (포맷 버그).
- 시나리오 8(규모 vs 규모)처럼 우연히 종류가 맞아떨어지는 경우엔 정상 판정되는 걸 보면, 산술 비교 로직 자체보다 **"비교 대상이 같은 종류인지 먼저 확인"하는 단계가 빠진 게 근본 원인**. `claim.claim_type`(규모/증감률/비교/전망)과 `computed.calc_type`(단순조회/증감/증감률/합계/비율)이 서로 다르면 애초에 코드 규칙 1차 필터를 적용하면 안 되고, "판단불가"로 넘기거나 종류를 맞추는 전처리가 필요함. D 담당 파일(`agent/verdict/judge.py`)이라 B가 직접 고치지 않고 이슈로만 남김 — 우선순위 높음, D에게 전달 필요.
