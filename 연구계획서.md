# 연구 계획서

## AI 기반 뉴스 사실 검증 시스템 설계 및 프로토타입 개발

> 뉴스 기사 내 수치 기반 주장을 탐지하고, KOSIS Open API 기반 공식 통계와 비교하여 사실 여부를 검증하는 AI 시스템 PoC

## Background & Baseline

### 관련 연구

| 연구/개념 | 핵심 내용 | 이 프로젝트와의 연결점 |
| --- | --- | --- |
| 자동 사실검증 3단계 파이프라인 | claim detection → evidence retrieval → verdict prediction(NLI 기반) | 검증 프로세스의 기본 골격 |
| FEVEROUS | 위키 기반 사실검증 데이터셋. 수치 주장은 전체의 약 10%뿐 | 수치 주장이 상대적으로 덜 연구된 영역임을 보여줌 |
| QuanTemp | 실제 발화 15,000여 건의 수치·시간 주장 벤치마크 | 수치 주장 전용 연구 공백을 메우려는 최근 흐름 |
| TabFact | 표 1.6만 개 근거, 11만 문장 ENTAILED/REFUTED 라벨 | 정형 데이터(표) 기반 검증의 대표 데이터셋 |
| Table-BERT (신경망) vs LPA (기호적) | 표+문장 직접 인코딩 vs 문장을 프로그램으로 파싱해 표에서 직접 연산 | "연산은 코드로" 택한 근거 계열 |
| TabVer | 자연 논리 규칙 + LLM 하이브리드 | 기호적 설명가능성 + 신경망 성능을 동시에 노림 |
| **"The Data Says Otherwise"** | Data Claim Detection → Text-to-data Mapping → Veracity Determination | **검증 프로세스와 구조적으로 가장 유사** |
| LLM 수치 추론 취약성 (최근 연구) | LLM은 겉보기와 달리 비교·산술·복합 수치 추론에 체계적 약점 | **"연산을 LLM에 맡길지 분리할지"** 설계 결정의 핵심 근거 |

### 이번 프로젝트의 Baseline

**1차 baseline: LLM End-to-End 프롬프팅**
- 기사 문장 + KOSIS 검색 결과(후보 통계표)를 통째로 프롬프트에 투입
- 일치/불일치/판단불가 판정까지 HCX 계열 LLM에게 한 번에 위임

**선정 이유**
1. 제공 리소스(HCX-DASH, HCX-003/005/007)만으로 빠르게 구현 가능 → 공정한 비교 기준선
2. 문헌상 지적된 "LLM 수치 추론 취약성"이 **우리 데이터에서도 실제로 재현되는지** 확인하는 참고선(reference line) 역할

## Proposed Method

### Problem Definition

| 구분 | 내용 |
| --- | --- |
| **Task** | 구조화된 공식 통계(KOSIS)를 근거로 한 수치 기반 뉴스 주장 검증 |
| **해결 문제** | 뉴스 인용 통계 수치가 KOSIS와 실제로 일치하는지, 혹은 시점·단위·모집단 차이에서 비롯된 착시적 불일치인지 자동 구분 |
| **왜 중요한가** | 왜곡된 통계·맥락 없는 인용·시점 단위 혼동·부분 발췌 오류가 반복되지만 수작업 검증 의존도가 높음 → 자동화 시 검증 커버리지·대응 속도 대폭 향상 |

### 핵심 아이디어 — Neuro-symbolic 4단계 파이프라인

```
① 추출          ② 매핑              ③ 연산                ④ 설명
문장→구조화  →  임베딩+리랭커로  →  API 조회 + 코드로   →  LLM으로
(지표/시점/     통계표 검색           직접 계산            판정 근거
 단위/모집단)                        (합계·비율·증감률)     설명 생성
```

- **① 주장 탐지·구조화 추출**: 지표명·시점·단위·모집단·비교유형 필드로 구조화
- **② KOSIS 표 매핑**: 임베딩+리랭커로 후보 검색 (실전1 통계표 요약을 pool로 활용)
- **③ API 조회 및 심볼릭 연산**: 실제 수치는 API로 조회, 계산은 **코드로만** (LLM에 산술 미위임)
- **④ 판정 및 설명 생성**: 계산값과 기사 수치 비교 → 일치/불일치/판단불가 + LLM 설명

### Baseline 대비 차이점

| 구분 | Baseline (LLM End-to-End) | Proposed (Neuro-symbolic) |
| --- | --- | --- |
| 구조 | 단일 LLM이 추출 ~ 연산 ~ 판정 전부 | 단계별 분리 — 연산만 결정론적 코드 |
| 수치 연산 신뢰도 | LLM 산술 오류 가능성 | 코드 연산 → 오류 없음 |
| 매핑 근거 | 프롬프트 컨텍스트에 암묵 의존 | 임베딩+리랭커 명시적 검색, 근거 제시 가능 |
| 설명 가능성 | 판정 과정 블랙박스 | 단계별 산출물 로그로 추적 가능 |

### Dataset & Preprocessing

**데이터셋**
- 조선일보 기사(2025.01.01~12.31) — **7/10 파일로 데이터셋 제공받음**
- 구성: 제목 / 작성일 / 섹션 / URL / 본문 / 키워드 레이블(True="국가데이터처"·"통계청" 검색, False=전체)
- KOSIS Open API 4종: 통계목록 / 통계자료 / 메타정보 / 통계설명
- 참고: 7/10 수동 검색·수집은 정식 데이터 수집이 아니라 공식 데이터셋 받기 전 감 잡기용 워밍업이었음

**데이터 특성 (예상)**
- keyword=True 그룹이 수치·통계 인용 밀도 높을 것 / False 그룹은 낮을 것
- 카테고리(인구/물가지수/범죄/소득계층)별 검증 가능 주장 비율·매핑 성공률 상이할 것 (7/10 인구 카테고리 EDA에서 KOSIS 표 5종 확인)

**전처리**
1. 문장 분리·정제, 수치·비교 표현 패턴으로 후보 필터링
2. 시점 표현("올해","지난해","상반기") 정규화
3. 카테고리별 KOSIS 표 요약 사전 구축 (표이름/분류항목/단위/시점범위/tblId)

### 가설 (왜 효과가 있을 것으로 예상하는가)

| 가설 | 내용 |
| --- | --- |
| **H1** | 연산을 코드로 분리 → LLM 수치 추론 취약성 회피 → baseline보다 정확도↑ |
| **H2** | 임베딩+리랭커 명시적 매핑 → 관련 없는 표 근거 오판정↓ → 판단불가·불일치 오분류↓ |
| **H3** | 단계별 산출물 로그 → 오류 원인(추출/매핑/연산) 구분 가능 → 개선 우선순위를 데이터 기반으로 결정 |

## Experiment Design

### 실험 목록

| 실험 | 내용 |
| --- | --- |
| E1 | Baseline(LLM End-to-End) 성능 측정 |
| E2 | Proposed(Neuro-symbolic) 성능 측정 |
| E3 | Ablation — 리랭커 유무 / symbolic 연산 유무 |
| E4 | 카테고리별(인구/물가지수/범죄/소득계층) 성능 비교 |
| E5 | 정성 분석 — 오분류·판단불가를 갭 유형(이름/시점표기/모집단·기준/출처/국내통계 有無)으로 분류 |

### 비교 대상
- Baseline vs Proposed (주 비교)
- 카테고리별 성능
- keyword True/False 그룹 간 검증 가능 주장 비율

### Evaluation Metrics

| 지표 | 측정 방법 |
| --- | --- |
| 판정 정확도 | 일치/불일치/판단불가 3-class Accuracy + Macro-F1 |
| 매핑 성능 | 올바른 표가 상위 K개 후보 내 포함되는지 Recall@K |
| 사람 평가 | 무작위 샘플의 사람↔모델 판정 일치도(agreement) |
| 정성 지표 | 갭 유형별 빈도, 판단불가 과다 여부(recall 저하 확인) |
| sanity check | 계산값을 KOSIS 원자료와 수동 대조 |
